## Описание проекта

В основе геймплея — задача продержаться 5 минут против нарастающих волн геометрических врагов. Игрок уничтожает цели, собирает кристаллы опыта (XP) и прокачивает характеристики персонажа при повышении уровня.

### Игровые механики
* **Выживание на время**: Матч длится фиксированные 5 минут с постепенным увеличением сложности.
* **Прогрессия**: Из врагов выпадают кристаллы опыта, которые заполняют шкалу уровня и позволяют улучшать статы.
* **Разновидности врагов**: В прототипе реализовано 4 вида противников со своими параметрами.
* **Управление**: Движение на WASD или стрелочки. Прицеливание и стрельба автоматические (оружие атакует само, когда враг оказывается в зоне видимости).

## Архитектура и паттерны

Проект написан по принципам SOLID со слабой связанностью компонентов, без использования тяжелых обновлений в Update для каждого объекта.

### Инструменты и паттерны
* **Zenject**: отвечает за биндинги, граф зависимостей и внедрение их в системы.
* **MVP**: используется для реализации UI. Презентеры написаны на чистом C#, они управляют логикой окон и подписками, оставляя View пассивными.
* **FSM**: управляет состояниями самой игры (главное меню, игровой процесс, экран улучшения, победа/поражение).
* **Strategy**: применяется для калькуляции прокачки статов персонажа, позволяя легко менять формулы прироста характеристик.
* **Object Pool**: реализован на основе очередей для переиспользования игровых объектов (врагов, пуль, кристаллов опыта) без просадок FPS и лишней сборки мусора (Zero-GC).
* **Signal Bus**: используется для глобального обмена сообщениями между независимыми системами (например, для уведомления UI о получении опыта, изменении здоровья игрока или триггере окончания матча).

### Оптимизация логики
* Поведение врагов, кристаллов опыта и пуль вынесено из MonoBehaviour-скриптов в отдельные менеджеры, которые обрабатывают объекты централизованно.
* Для логики регистрации попаданий пуль написан легковесный самописный сервис, заменяющий тяжелые стандартные триггеры.
* Компоненты `Rigidbody` используются на объектах исключительно как изолированное решение, чтобы сущности физически не входили друг в друга и не слипались в одну точку. Вся остальная симуляция и перемещение просчитываются кодом.
